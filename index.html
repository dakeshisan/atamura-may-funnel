Полный путь клиента: от лида до встречи и брони

Май 2026. Сквозная воронка по двум воронкам Bitrix (Телемаркетинг → Отдел продаж) и воронке покупателей — что удаётся собрать достоверно, а что рвёт сам CRM.

Источник: Bitrix24 live · метод: история переходов по стадиям (crm.stagehistory), сшивка по ID сделки · 23 944 события · сгенерировано 16.06.2026
Короткий ответ. Полную воронку «лид → встреча в отделе продаж» построить можно — и поток по всем стадиям Bitrix за май считается надёжно (панель 1). Но сквозную воронку по одному клиенту от лида до брони достоверно собрать сегодня нельзя: сам CRM рвёт нить клиента на стыках воронок (при передаче в продажи часто заводится новая сделка с другим ID; бронь фиксируется в отдельной «Воронке купивших» и сшивается по ID лишь наполовину). Это не дефект аналитики — это дефект данных и процесса линковки в Bitrix, и это главный вывод. Цифры ниже — для диагностики «где теряем», не для KPI/депремирования.

Что собирается достоверно, а что — нет

✓ Надёжно

  • Поток за месяц по каждой стадии обеих воронок — реальные счётчики входов в стадию (панель 1).
  • Форма потерь: где массово копятся отказы и недозвоны.
  • Объём встреч и броней за май как факт CRM.

✗ Недостоверно сегодня

  • Сквозная конверсия одного клиента лид→встреча→бронь — нить ID рвётся на хендоффах.
  • Вклад Телемаркетинга в продажи — большинство встреч в ОП приходит «не из ТМ» по ID.
  • Привязка брони к источнику лида — покупатели в отдельной воронке, сшивка ½.

Как читать. Самый резкий обрыв — на входе: 87,5% лидов остаются в статусе «Неразобранное» и не сдвигаются по стадиям. Вероятная причина — приток (9 700/мес) сильно превышает пропускную способность отдела, лиды копятся в NEW. Доходимость до брони по ID = 3 сделки — это артефакт обрыва нити (бронь под другим ID), а не реальные продажи мая (их 50, см. панель 1). Подкогорта 1–20 мая (зрелая, n=6 463) даёт ту же форму — правоцензурирование картину не объясняет.

Панель 3 · Где рвётся путь клиента

Четыре измеренных дефекта данных, из-за которых сквозная воронка по клиенту сегодня недостоверна. Это и есть список «что чинить в Bitrix».

1. Хендофф ТМ → ОП плодит новые сделки. Из лидов, получивших стадию «Записан на встречу», лишь 31% имеют сделку в Отделе продаж под тем же ID. Остальные — новые сделки с новым ID → исходный лид «зависает», его вклад теряется.
2. Большинство встреч в ОП — «не из ТМ» по ID. В мае в ОП проведено 554 встречи, но из майской когорты ТМ по ID дошло лишь 29. Встречи идут под другими ID / из бэклога прошлых месяцев.
3. Бронь в отдельной воронке, сшивка ½. Покупки фиксируются в «Воронке купивших» (cat 22, по ЖК). Из 21 покупателя мая по ID сделки сшивается 10 — связь «встреча → покупка» по клиенту восстанавливается лишь наполовину (для сравнения: по контакту в v2.0 — 0,2%, но это другой метод и период).
4. 87,5% лидов застревают в «Неразобранное». Либо лиды реально не обрабатываются (приток > ёмкости), либо работа не отражается сменой стадии. И то, и другое — повод навести дисциплину статусов.
Что нужно, чтобы воронка стала сквозной (приоритет):
  • Связать сделки между воронками одним ключом (сохранять ID или единый client-key при передаче ТМ→ОП→Купившие) — без этого сквозная конверсия неизмерима.
  • Дедупликация контактов в Bitrix (первопричина «покупателей без следа»).
  • Дисциплина статусов: «Взято в работу» ставить при первом контакте, не оставлять в NEW.
  • После фикса — пересобрать эту воронку как когортную (она уже автоматизирована).

Оговорки (важно при чтении)

  • Стадия ≠ факт. «Встреча проведена» / «Бронь» — это отметка менеджера в CRM, а не подтверждённое физическое событие. Возможны оптимистичные/пакетные простановки.
  • Это диагностика, не KPI. Цифры для понимания «где теряем», не для депремирования менеджеров (риск по ТК РК; «встреча» по CRM — прокси).
  • Май, по часовому поясу портала (Bitrix отдаёт +03:00). Всплеск 14 мая (1 246 лидов) — похоже на пакетный импорт.
  • Когорта по ID — нижняя граница. Из-за обрыва нити (панель 3) доходимость занижена; верхняя оценка — в потоке (панель 1).
  • Хвост (бронь/выручка) до фикса линковки не трактовать как эффективность продаж.
